**Ticket: Improve Element Insertion Flow (Reduce Friction, Increase Writing Speed)**

---

### Problem

URL: https://insanely-great-writer-holy-fog-4587.fly.dev/docs/1
URL (Local): http://localhost:3000/docs/1

Current flow is too manual:

1. Select element type (bottom-left dropdown)
2. Click **Insert**
3. Type content
4. Press Enter
5. Repeat entire process

This breaks writing flow. It feels like using a form, not writing.

Goal: make this behave closer to **Final Draft / Notion / Vim-like fluid typing**, where structure follows intent, not clicks.

---

## Proposed Solutions (ranked by complexity vs impact)

---

### 1. **Auto-advance element type on Enter (Low effort, high ROI)**

**Behavior**

* User selects an element once (e.g. Scene Heading)
* Presses Enter → system auto-inserts the *next logical element*
* No need to re-click dropdown or Insert

**Example flow**

* Scene Heading → Enter → Action
* Action → Enter → Action (default repeat)
* Character → Enter → Dialogue
* Dialogue → Enter → Character

**Why this works**

* Mirrors industry tools (Final Draft)
* Removes 50–70% of clicks immediately
* Keeps UI mostly unchanged

**Implementation idea**

* Maintain a simple state machine:

```ts
const NEXT_ELEMENT = {
  scene_heading: "action",
  action: "action",
  character: "dialogue",
  dialogue: "character",
  parenthetical: "dialogue"
};
```

* On `Enter`:

  * Insert new block
  * Set its type = `NEXT_ELEMENT[currentType]`

---

### 2. **Keyboard-first insertion (Medium effort, high ROI)**

**Behavior**

* User never touches dropdown
* Typing patterns define element type

**Examples**

* `INT.` → auto Scene Heading
* `JOHN` (uppercase line) → Character
* After Character → auto Dialogue
* `(` → Parenthetical
* `>` or `CUT TO:` → Transition

**Why this works**

* Removes UI dependency entirely
* Feels “smart” and fast
* Aligns with real screenplay writing habits

**Implementation idea**

* On line commit (Enter):

  * Run lightweight parser:

```ts
function detectType(text: string): ElementType {
  if (text.match(/^(INT\.|EXT\.)/)) return "scene_heading";
  if (text === text.toUpperCase() && text.length < 30) return "character";
  if (text.startsWith("(")) return "parenthetical";
  return "action";
}
```

* Override current block type if confidence is high

---

### 3. **Slash command menu (`/`) (Medium effort, flexible UX)**

**Behavior**

* Typing `/` opens inline command palette
* User selects element quickly

**Example**

```
/sc → Scene Heading  
/ch → Character  
/di → Dialogue
```

**Why this works**

* Familiar (Notion, Slack)
* Works well for new users
* Doesn’t force full automation

**Implementation idea**

* On `/` key:

  * Show floating menu near cursor
  * Filter options by typed string
  * On select → convert current block type

---

### 4. **Inline element switching via Tab / Shift+Tab (Medium effort)**

**Behavior**

* Tab cycles forward through element types
* Shift+Tab cycles backward

**Why this works**

* Fast correction without mouse
* Keeps user in flow

**Implementation idea**

```ts
const ELEMENT_ORDER = [
  "scene_heading",
  "action",
  "character",
  "dialogue",
  "parenthetical",
  "transition"
];
```

* Maintain index per block
* Tab increments index

---

### 5. **Remove “Insert” button entirely (Higher impact change)**

**Behavior**

* Every line is already an element
* Typing defines it, not insertion

**Why this works**

* This is the real fix
* “Insert” is a legacy UI pattern
* Modern editors treat everything as a block

**Implementation idea**

* Model document as array of blocks from start
* On Enter:

  * Always create next block
  * Assign type via:

    * previous type (state machine)
    * or parser (from Solution #2)

---

## Recommendation (practical path)

Start with this sequence:

1. **Auto-advance on Enter**
2. Add **Tab switching**
3. Add **lightweight parser (INT., uppercase detection)**

Skip slash menu for now unless needed.

Do not remove Insert yet, but make it optional.

---

## Success Criteria

* User can write 10 lines without touching mouse
* Element switching happens without explicit clicks
* Time-to-write-first-page drops significantly
* UI feels invisible during writing

---

## Risk / Trade-offs

* Over-aggressive auto-detection can frustrate users
* Need clear override (Tab or manual switch)
* Must keep behavior predictable

---

## Bottom line

Right now this is a UI-driven editor.
You want a **flow-driven editor**.

The fastest win is simple:

> Press Enter → system decides what comes next.
